## Purpose

Define a `qianji`-first workflow where documentation authoring is governed by the `xiuxian-testing` contract kernel, so docs work is not just prose generation but an auditable engineering loop.

The primary entry point is the existing `qianji` scenario audit lane (`formal_audit` task type), extended with contract-feedback projection.

## External Reference Pattern (Comparative Only)

`Ralph-Workflow` treats documentation as a first-class unattended task:

- work starts from `PROMPT.md` (often generated by a docs work guide),
- planning produces `.agent/PLAN.md`,
- development and analysis iterate until plan satisfaction,
- review and fix loops run before final commit.

For our V1, these primitives are comparative references. The production orchestrator remains `qianji`.

## Xiuxian Mapping

The mapping below keeps ownership boundaries explicit:

| Workflow stage                | System of record                 | Xiuxian component                                      |
| ----------------------------- | -------------------------------- | ------------------------------------------------------ |
| task intent and acceptance    | product requirement envelope     | `qianji` workflow input                                |
| planning baseline             | iteration plan context           | `qianji` workflow state                                |
| deterministic contract checks | pass/fail/warn contract evidence | `xiuxian-testing` rule packs and `ContractSuiteRunner` |
| advisory role audit           | role-attributed interpretation   | `qianhuan` + `qianji` (+ `zhenfa` stream)              |
| persisted audit memory        | trace and finding retrieval      | `wendao` via contract knowledge export                 |

## Execution Contract

The V1 execution order is strict:

1. Use `qianji` workflow orchestration to generate docs intent and plan context.
2. Run deterministic contract packs in `xiuxian-testing` against docs and API artifacts.
3. Trigger advisory audits only when policy thresholds are met.
4. Merge deterministic and advisory findings into one `ContractReport`.
5. Export `ContractKnowledgeBatch` to Wendao-facing storage for longitudinal retrieval.

Deterministic checks remain the source of truth for policy gates. Advisory roles provide interpretation and remediation depth.

## Required Contract Surfaces

The workflow bridge must preserve these surfaces:

- `ContractSuiteRunner` as the deterministic + advisory merge boundary.
- `AdvisoryAuditRequest` as the cross-system handoff payload.
- `ContractKnowledgeBatch` as the stable export envelope for Wendao ingestion.
- role provenance (`advisory_role_ids`) and `trace_ids` for audit replayability.

## V1 Command Shape

This project does not require one specific orchestrator binary. A representative unattended flow is:

```bash
# Contract kernel validates artifacts managed by qianji workflow
cargo test -p xiuxian-testing --test contracts_rest_docs
cargo test -p xiuxian-testing --test contracts_runner
cargo test -p xiuxian-testing --test contracts_knowledge_export
```

The key principle is: `qianji` is the primary engine; external workflow tools are optional adapters.

## Scenario Audit Binding

In runtime terms, this alignment binds directly to the `qianji` scenario audit path:

- `formal_audit` mechanism executes advisory policy and retry control.
- `run_contract_feedback_flow` runs suite evaluation and advisory merge.
- `run_and_persist_contract_feedback_flow` projects findings into Wendao-ready entries through the sovereign sink.

## Non-Goals

- No workflow engine implementation inside `xiuxian-testing`.
- No replacement of deterministic rules with LLM-only review.
- No direct pass/fail authority delegated to Wendao persistence.
