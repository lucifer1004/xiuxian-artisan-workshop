### [IN-PROGRESS] Feature: Wendao Repo Intelligence Source Lifecycle and Sync Control

- **Objective**: Move Repo Intelligence repository sourcing into an explicit common-core lifecycle that validates local checkouts, prepares cache-local mirrors for remote URLs, materializes managed working checkouts, and exposes that state through a dedicated sync control surface.
- **Status**: [In Progress] `xiuxian-wendao::repo_intelligence` now supports three repository source modes behind the same query surface: validated local checkout paths, remote `url` plus optional `ref`, and explicit managed-checkout refresh policy via `refresh = "fetch" | "manual"`. Remote repositories are now materialized through a two-stage cache layout under `PRJ_CACHE_HOME/xiuxian-wendao/repo-intelligence/`: a bare mirror in `mirrors/<repo-id>.git` followed by a working checkout in `repos/<repo-id>`. The checkout tracks the cache-local mirror as `origin`, while repository records still expose the upstream URL from configuration plus the active checkout revision. The common core now also exposes `wendao repo sync --repo <id>` and the matching Studio gateway route `GET /api/repo/sync?repo=<id>&mode=<ensure|refresh|status>`, both of which report the resolved source kind, requested sync mode, configured refresh policy, mirror/check-out lifecycle states, observation time (`checked_at`), last local mirror fetch time (`last_fetched_at`), mirror revision, tracking revision, drift state, high-level `health_state`, freshness-oriented `staleness_state`, grouped `status_summary`, checkout path, optional mirror path, upstream URL, and revision without forcing a full repository analysis pass. `--mode refresh` overrides `refresh = "manual"` for the current sync operation, while `--mode status` is read-only and reports `missing` or `observed` lifecycle states without creating mirrors, creating checkouts, or touching the network. Managed remote sync now also reports a finer local drift signal through `mirror_revision + tracking_revision + drift_state`, distinguishing `ahead`, `behind`, and `diverged` cases instead of a single coarse mismatch state, plus a compact health summary through `healthy`, `missing_assets`, `needs_refresh`, `has_local_commits`, `diverged`, and `unknown`, a mirror freshness summary through `fresh`, `aging`, and `stale`, and a grouped status view that bundles lifecycle readiness, freshness, revision alignment, and attention signaling for agents. ### [IN-PROGRESS] Feature: Wendao External Modelica Repo Intelligence Boundary

- **Objective**: Validate that Repo Intelligence stays Wendao-native in the common core while allowing an external Rust crate to implement Modelica/MSL-specific analysis against the same registry, record, and query surface.
- **Status**: [In Progress] `xiuxian-wendao` now exposes registry-aware library entry points for `repo.overview`, `module.search`, `symbol.search`, `example.search`, and `doc.coverage`, so external crates can reuse the same configured query surface with custom plugin registries instead of depending on CLI-only entry points. The workspace now includes `packages/rust/crates/xiuxian-wendao-modelica`, which registers `plugins = ["modelica"]` through `register_into(...)` and currently performs conservative Modelica repository analysis from `package.mo` layout, lightweight `.mo` declaration scanning, `Examples`, `UsersGuide`, and inline `annotation(Documentation(...))` docs, emitting normalized `RepositoryRecord`, `ModuleRecord`, `SymbolRecord`, `ExampleRecord`, `DocRecord`, plus `Contains`, `Declares`, `ExampleOf`, and `Documents` relations. The external walker now skips hidden/VCS paths, so repository internals no longer leak into doc inventory. The crate has been modularized into a `plugin/` feature folder, `package.order` now feeds canonical module and example ordering through the common-core query surface because equal-score `module.search` and `example.search` matches preserve analyzer order instead of being force-sorted alphabetically, discovery now classifies repository paths into API/example/documentation/support surfaces so runnable `Examples/` models stay in the example surface while `Examples/ExampleUtilities`, `Examples/.../Utilities`, `Internal`, and `UsersGuide/` stay out of the default symbol and example surfaces, support-only package modules now also stay out of the default module surface, the relation layer now links both `UsersGuide` file docs and `UsersGuide` annotation docs back to their owning functional modules plus visible `UsersGuide` hierarchy modules so module-scoped `doc.coverage` can surface nested guide pages directly, `UsersGuide` docs now also emit semantic `DocRecord.format` hints for `Tutorial`, `Conventions`, `Connectors`, `Implementation`, `RevisionHistory`, `VersionManagement`, `ReleaseNotes`, `References/Literature`, `Overview`, `Contact`, `Glossar/Glossary`, `Concept/*Concept`, and `Parameters/Parameterization` content with matching `_annotation` variants for inline documentation payloads, `Conventions.mo` can now also synthesize subsection docs for stable nested topics (`Documentation`, `ModelicaCode`, `Icons`) when the nested declaration block carries its own `annotation(Documentation(...))`, `ReleaseNotes.mo` can now also synthesize subsection docs for stable nested topics (`VersionManagement` and annotated `Version_*` entries), synthetic subsection docs now normalize `DocRecord.title` into projection-ready labels while preserving stable section identities in `doc_id` and `path`, and the same signals are now smoke-tested through Wendao's deterministic `ProjectionInputBundle` contract instead of staying limited to raw repo inventory snapshots. The projection handoff now also preserves shared-target context: module reference seeds can absorb child-symbol docs and examples, example how-to seeds can carry direct target-side docs and format hints, guide-doc seeds can carry direct target-side examples, and symbol-targeted annotation docs now classify as `reference` seeds instead of generic explanation seeds. ### [IN-PROGRESS] Feature: Sentinel Docs Governance for Package-Local Crate Docs

- **Objective**: Extend the existing Wendao Sentinel audit/fix pipeline so package-local crate docs can be governed automatically instead of relying on ad-hoc manual cleanup.
- **Status**: [In Progress] `xiuxian-wendao::zhenfa_router::native::semantic_check` now includes eleven docs-governance slices for package-local crate docs. The first rule inspects existing package-doc files and emits `doc_identity_protocol` issues when the primary `:ID:` is human-readable instead of opaque, or when the top-level property drawer is missing an `:ID:` line. The second rule runs at workspace scope and emits `missing_package_docs_index` when a crate already has a package-local `docs/` tree but lacks `docs/index.md`. The third rule also runs at workspace scope and emits warning-level `missing_package_docs_tree` when a Rust crate with `Cargo.toml` has no package-local `docs/` directory at all. The fourth rule emits warning-level `missing_package_docs_section_landing` when a crate already has `docs/index.md` but is still missing one or more standard section landing pages under `01_core`, `03_features`, `05_research`, or `06_roadmap`. The fifth rule emits warning-level `missing_package_docs_index_section_link` when `docs/index.md` omits the standard section link for an already-existing landing page. Section-link insertion now recognizes decorated headings such as `## 01_core: ...` and, when the heading is genuinely absent, inserts before `:RELATIONS:` or footer metadata instead of appending a new block after the footer. The sixth rule emits warning-level `missing_package_docs_index_relations_block` when the package-index body already contains links but the bottom `:RELATIONS: :LINKS:` block is still absent. The seventh rule emits warning-level `missing_package_docs_index_relation_link` when the package-index body already contains links that the bottom `:RELATIONS: :LINKS:` block omitted. The eighth rule emits warning-level `stale_package_docs_index_relation_link` when the bottom `:RELATIONS: :LINKS:` block still contains extra links that no longer exist in the package-index body. The ninth rule emits warning-level `missing_package_docs_index_footer_block` when `docs/index.md` already has a bottom relations block but still lacks the trailing `:FOOTER:` metadata block. The tenth rule emits warning-level `incomplete_package_docs_index_footer_block` when `docs/index.md` already has a footer block but is still missing required fields such as `:STANDARDS:` or `:LAST_SYNC:`. The eleventh rule now emits warning-level `stale_package_docs_index_footer_standards` when `docs/index.md` already has a complete footer block but still carries an older `:STANDARDS:` value such as `v1.0`. Workspace docs-governance now also recurses through package-local docs trees and emits the same `doc_identity_protocol` issues for nested package docs instead of only enforcing opaque `:ID:` values during explicit single-file audits. Surgical fix generation now supports all eleven issue types directly: `doc_identity_protocol`, `missing_package_docs_index_section_link`, `missing_package_docs_index_relations_block`, `missing_package_docs_index_footer_block`, `incomplete_package_docs_index_footer_block`, `stale_package_docs_index_footer_standards`, `missing_package_docs_index_relation_link`, and `stale_package_docs_index_relation_link` use surgical byte-range insertions or replacements, while bootstrap-style asset creation still rides `BatchFixMode::CreateFile` through `AtomicFixBatch`. `run_audit_core` now also seeds explicit workspace doc targets into `file_contents`, which closes the last execution gap between workspace-issued docs-governance findings and `wendao fix`; workspace-issued package-doc identity findings are now seeded back into `file_contents` as well, so package-doc directory audits keep the surgical fix path alive; explicit package-doc audits now also execute direct document-governance checks for the requested file; package-doc scope matching now uses path-aware ancestor/descendant checks instead of raw substring matching, which prevents `xiuxian-wendao/docs` and `xiuxian-wendao-modelica/docs` from bleeding into each other; and audit summary/file-report rendering now deduplicates alias doc paths so explicit workspace audits report one physical `docs/index.md` instead of separate absolute and relative entries. That hardened path has already been exercised on real package docs: `xiuxian-llm/docs/index.md` and `xiuxian-testing/docs/index.md` were remediated through the same audit/fix pipeline until both package indexes returned audit-clean reports with opaque top-level `:ID:` values. The package-index renderer now also emits the footer metadata block by default, so index bootstrap and index repair no longer diverge on footer shape, existing footer blocks can be normalized in place when required metadata fields are missing, and older footer standards versions can be normalized back to `v2.0` without losing `:LAST_SYNC:`. The widened workspace scan is also now verified against real package-doc directory scopes: `wendao audit packages/rust/crates/xiuxian-wendao/docs` reports the expected legacy `ERR_DOC_IDENTITY_PROTOCOL` set across Wendao’s package-doc tree, while `wendao audit packages/rust/crates/xiuxian-wendao-modelica/docs` stays isolated and reports zero top-level package-doc identity errors. There is now also a targeted end-to-end regression proving that one package-doc directory-scope audit/fix batch can rewrite multiple nested `doc_identity_protocol` violations atomically, instead of relying on repeated single-file `wendao fix` calls. This means the current Sentinel stack can now rewrite or insert package-doc `:ID:` values, materialize a missing package-local docs index, bootstrap a missing package-local docs tree by creating `docs/index.md`, create the standard section landing pages, stitch those pages back into the package index, bootstrap a missing bottom relations block, bootstrap a missing footer block, normalize an incomplete footer block, normalize stale footer standards drift, and then keep that bottom relations drawer aligned with the body links in both directions through the same audit/fix flow without adding a separate docs command. Full directory scaffolding beyond those standard landing pages and coherence rules is still out of scope for this slice.
