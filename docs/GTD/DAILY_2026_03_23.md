### [IN-PROGRESS] Feature: Studio Search Plane Reconstruction

- **Objective**: Keep moving Studio search off request-path scans and onto the Lance/Arrow/Valkey search plane, with repo-backed reads and reranking bounded by combined repo-corpus state instead of transient repo-index phases or full batch materialization.
- **Status**: [Tier-3 Gate Green] The governing execution record remains `[[.cache/codex/execplans/wendao-studio-search-plane.md]]`, and the package roadmap sync remains `[[packages/rust/crates/xiuxian-wendao/docs/06_roadmap/406_studio_search_plane.md]]`.
- **Update**: Repo-backed publication manifests are now gone. Combined `SearchRepoCorpusRecord` / `SearchRepoCorpusSnapshotRecord` state is the only persisted runtime and publication recovery surface for `repo_entity` and `repo_content_chunk`, and human-readable status text now consistently refers to published state.
- **Update**: `xiuxian-vector` now exposes streaming columnar scan and FTS consumers, so repo-backed search can process projected Lance batches incrementally instead of forcing `Vec<RecordBatch>` materialization on the hot path.
- **Update**: `repo_entity` and `repo_content_chunk` now rerank in bounded streaming windows. Repo-backed search keeps only capped candidate/path working sets while batches stream in, which lowers peak memory and keeps wide repo queries from scaling with full recall volume.
- **Update**: `knowledge_section` and `attachment` now follow the same streaming discipline. Both query paths consume Lance batches incrementally, and `knowledge_section` also trims its path-dedup working set so note-heavy queries no longer scale with full batch recall volume.
- **Update**: `search_plane::ranking` now centralizes the bounded rerank window mechanics used by `repo_entity`, `repo_content_chunk`, `knowledge_section`, and `attachment`. The active query lanes now share one trim/sort/window policy instead of open-coding their own retained-target and threshold logic.
- **Update**: The repo analyzer fast-path is repaired and back on the formal gate. `analyzers::service::search::artifacts` now owns cached `RepositorySearchArtifacts` construction, projected-page artifact search is wired back through the current projection module surface, and the stale duplicate helper in `documents.rs` is removed.
- **Update**: Repo-backed query cache keys now keep the publication identity for stable ready reads, but they also append repo phase plus current/published revision fragments when a refresh is active or a ready repo has drifted away from the published revision. This preserves warm-cache wins without hiding refresh-state or mismatch-state responses behind stale ready-cache entries.
- **Update**: `/api/search/index/status` now flags ready-state revision mismatches. When a repo reports `ready` but the published manifest targets a different source revision, the repo-backed corpus stays readable yet exposes an explicit consistency error instead of silently claiming the manifest matches the current repo revision.
- **Update**: Search-plane lifecycle now has an explicit `degraded` phase for readable-but-inconsistent corpora. Repo-backed status synthesis upgrades to `degraded` when published tables still serve reads but manifest drift or partial repo failures are present, instead of overloading plain `ready`.
- **Update**: Studio `/api/search/index/status` now exposes `degraded` in both the per-corpus phase and the aggregate counters, so clients can distinguish "fully ready" from "serving stale/partial repo-backed data" without scraping `lastError`.
- **Update**: Repo-backed status now emits machine-readable `issues` alongside `last_error`. Each issue carries a stable code, `repo_id`, readability flag, and revision metadata, so the UI can branch on `published_manifest_missing`, `published_revision_missing`, `published_revision_mismatch`, and `repo_index_failed` without parsing human strings.
- **Update**: Each corpus status row now also exposes `issueSummary`, which folds the raw issue list into `family + primaryCode + issueCount + readableIssueCount`. Clients can render a stable badge or severity lane first, then expand the detailed issue list only when needed.
- **Update**: Each corpus status row now also exposes `statusReason`, which folds lifecycle phase plus issue state into one direct UI decision surface: `code + severity + action + readable`. Initial indexing now reports `warming_up`, refresh indexing reports `refreshing`, failed rebuilds report `build_failed`, and repo-backed consistency failures map directly to repo repair or sync inspection actions.
- **Update**: `statusReason` now also absorbs maintenance state for healthy readable corpora. A ready corpus with queued background compaction now reports `compaction_pending + info + wait + readable`, so the UI can show search-plane maintenance without reverse-engineering `maintenance.compactionPending`.
- **Update**: `/api/search/index/status` now also exposes a response-level `statusReason` summary. It selects the dominant corpus reason by `severity -> code` priority and carries `affectedCorpusCount`, `readableCorpusCount`, and `blockingCorpusCount`, so clients can render one top-level state banner without rescanning all corpus rows.
- **Update**: Maintenance telemetry now distinguishes queued compaction from actively running compaction. Corpus maintenance rows expose `compactionRunning`, per-corpus `statusReason` upgrades from `compaction_pending` to `compacting` while a compaction task is in flight, and the response-level aggregate `statusReason` now prefers `compacting` over `compaction_pending` at the same severity.
- **Update**: Repo-backed corpora now synthesize real epoch semantics instead of remaining epochless status rows. Published repo tables fold into a stable synthetic `active_epoch`, in-flight repo refresh work folds into a synthetic `staging_epoch`, and refresh no longer destroys the identity of the currently readable publication.
- **Update**: Repo publication manifests now persist their own `active_epoch`. `SearchRepoManifestRecord` derives and stores a stable epoch from `publication_id`, repo-backed status synthesis now aggregates those persisted epoch values instead of recomputing directly from publication strings, and legacy manifests keep working through a deterministic fallback path.
- **Update**: The formal verification lane for this slice is back on `cargo nextest`. The remaining blocker was unrelated `semantic_check` test visibility drift plus the new `valkey_common` env-helper tests; both were repaired so the repo-publication epoch slice can be validated without dropping back to plain `cargo test`.
- **Update**: Repo-backed corpora now reuse `SearchPlaneCoordinator` as their runtime state store instead of existing only as `status_with_repo_content()` overlays. The synthesized `RepoEntity` and `RepoContentChunk` status rows are now written back through coordinator state, so `service.status()` can return the last synchronized repo-backed `active_epoch`, `staging_epoch`, fingerprint, and row-count snapshot without requiring a fresh overlay step.
- **Update**: Repo-aware cache keys and handler read paths now consume synchronized repo runtime state instead of directly rebuilding version fragments from each `RepoIndexStatusResponse`. `code_search` and code-biased `intent` still sample repo-index status once per request, but they now push that state into `SearchPlaneService` and then read cache/phase decisions back from the search-plane runtime surface.
- **Update**: Repo search control flow now also uses search-plane semantics instead of open-coding repo-index phase checks in handlers. `StudioState::synchronize_repo_search_runtime(...)` is now the only handler-side bridge to repo-index status, `SearchPlaneService::repo_search_publication_state(...)` centralizes `entity/content/searchable/pending/skipped` classification, and both `code_search` and code-biased `intent` consume that single decision surface.
- **Update**: Repo runtime no longer needs request-path sampling for Studio search. `RepoIndexCoordinator::refresh_status_snapshot()` now pushes the authoritative aggregate snapshot into `SearchPlaneService::synchronize_repo_runtime(...)`, runtime synchronization replaces the full repo-runtime map instead of incrementally accreting stale entries, and the `code_search` / code-biased `intent` handlers no longer call into `repo_index.status_response(...)` just to refresh runtime state before building cache keys or pending/skipped decisions.
- **Update**: Repo runtime persistence is now folded into the combined repo-corpus cache layer. `SearchPlaneService::synchronize_repo_runtime(...)` persists refreshed `SearchRepoCorpusRecord` rows plus the combined `SearchRepoCorpusSnapshotRecord`, so runtime survives process-local memory loss without dual-writing standalone runtime keys.
- **Update**: Repo-aware read paths now hydrate runtime from persisted combined repo-corpus rows on memory miss. `repo_search_publication_state(...)`, repo-backed status synthesis, and repo-aware cache-key generation no longer reload standalone runtime rows from Valkey.
- **Update**: The old `SearchRepoRuntimeSnapshotRecord` path is gone. `SearchPlaneService::status_with_repo_runtime()` now synthesizes repo-backed corpus rows only from the search-plane-owned combined repo-corpus snapshot plus publication manifests, instead of accepting a fresh `RepoIndexStatusResponse` overlay.
- **Update**: Studio `/api/search/index/status` no longer reaches back into `repo_index.status_response(...)` to build repo-backed rows. The status endpoint now asks `SearchPlaneService` to refresh repo-backed corpus rows from its own runtime snapshot, so status, cache keys, and repo availability decisions all consume the same search-plane-owned runtime source.
- **Update**: Producer-side repo snapshot refresh now also keeps repo-backed corpus coordinator rows warm. `RepoIndexCoordinator::refresh_status_snapshot()` still pushes runtime into `SearchPlaneService`, and that push now fans out into repo-backed corpus status replacement as well as runtime persistence, so coordinator snapshots stay aligned without request-path overlays.
- **Update**: Repo runtime and repo publication now also share a persisted repo-backed row model. `SearchRepoCorpusRecord` stores `runtime + publication` for each repo-backed corpus row, and `SearchRepoCorpusSnapshotRecord` stores the full search-plane-owned repo-backed view for restart-stable hydration.
- **Update**: Repo-aware availability checks, repo-backed status synthesis, and repo-aware cache-key generation now read combined repo-corpus rows first instead of independently rebuilding state from runtime snapshots plus publication manifests.
- **Update**: Combined repo-corpus reads now reconcile against the freshest in-memory runtime/publication before answering the hot path. This removes the last async-persistence race where a repo refresh could keep serving the old cache identity until background cache writes completed.
- **Update**: `SearchPlaneService` no longer keeps separate in-memory `repo_runtime` / `repo_publications` side maps. Combined `repo_corpus_records` are now the only process-local repo-backed truth.
- **Update**: Legacy standalone `repo_runtime` Valkey rows and the old runtime-snapshot key are now fully removed. Restart hydration for repo-backed runtime now only flows through persisted combined `SearchRepoCorpusRecord` / `SearchRepoCorpusSnapshotRecord`; per-corpus publication manifests remain the narrow publication-only fallback beneath that combined read model.
- **Update**: That remaining publication-only fallback is now removed from repo-backed read paths. `repo_search_publication_state(...)`, repo-backed status synthesis, and repo-aware cache-key generation now require combined repo-corpus record/snapshot recovery and no longer self-heal from manifest-only Valkey state.
- **Update**: Producer-side runtime refresh now follows the same rule. `synchronize_repo_runtime(...)` no longer reattaches publication from persisted manifest rows when rebuilding repo-backed corpus rows after a memory miss; only persisted combined repo-corpus data can restore published state.
- **Update**: Repo publication manifests are now removed entirely. Repo-backed publication metadata lives only inside combined `SearchRepoCorpusRecord` / `SearchRepoCorpusSnapshotRecord` state as `SearchRepoPublicationRecord`, so there is no longer a separate repo-manifest keyspace or dual-write path to keep in sync.
