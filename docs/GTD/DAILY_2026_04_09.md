# GTD: Sovereign Nexus (2026-04-09)

## 1. [ACTIVE] Wendao DuckDB Bounded Analytics

- **Task**: Audit the residual architectural value of DataFusion after the
  DuckDB-first search execution direction was clarified.
- **Status**: [Residual Value Audit Landed] The current code and docs are now
  tightened around one narrower split: `WendaoArrow`, pyarrow, julia-arrow,
  and Flight own the cross-language Arrow `RecordBatch` substrate; DuckDB is
  the intended search-side database execution lane over Parquet and bounded
  request-scoped analytic relations; DataFusion retains only residual Rust-side
  live Arrow compute and migration-baseline value where the data is still a
  generated Arrow workset.
- **Follow-up**: [Residual Path Classification Landed] The remaining
  DataFusion references are now classified more concretely too. Request-scoped
  SQL, GraphQL planning, and Parquet search-plane reads are same-layer search
  execution residue that should migrate away, while only in-memory live Arrow
  compute and migration-baseline paths remain defensible retained value.
- **Follow-up**: [GraphQL SQL Translation Cutover Landed] The canonical
  `search/queries/graphql/` adapter no longer builds DataFusion expressions
  directly. It now keeps document parsing and argument decoding adapter-local,
  translates the resulting table query into SQL text, and executes through the
  shared request-scoped SQL seam, with focused translation and end-to-end
  GraphQL proofs.
- **Follow-up**: [Shared SQL Parquet Routing Landed] The canonical
  `search/queries/sql/` execution seam now reuses the same published-Parquet
  routing helper as FlightSQL for simple single-table queries over active
  local corpora and concrete repo publication source tables. Eligible SQL,
  FlightSQL, and GraphQL-to-SQL queries can now hit the same bounded
  `ParquetQueryEngine` lane, while discovery catalogs, logical views, and
  multi-source queries still stay on the shared SQL fallback.
- **Follow-up**: [Parquet Readability Naming Landed] Publication readiness
  helpers no longer describe Parquet corpora as `DataFusion`-readable. The
  code now treats repo and local publications as Parquet/query-engine
  readable, which matches the actual Parquet publication owner plus the
  bounded DuckDB/DataFusion execution split.
- **Follow-up**: [Search Engine Naming Cutover Landed] `SearchPlaneService`
  no longer exposes the retained DataFusion fallback as a generic
  `search_engine()` handle. The service boundary and the routed Parquet query
  callers now name that residual lane explicitly as
  `datafusion_query_engine()`, which matches the current architecture rather
  than implying that DataFusion still owns the generic search side.
- **Follow-up**: [DuckDB-Only Routed Parquet Cutover Landed] Eligible routed
  published-Parquet reads now stop accepting a production DataFusion fallback
  in `duckdb` builds. Repo gateway reads, local-corpus gateway reads, routed
  FlightSQL statements, and shared SQL single-table Parquet routing now
  select DuckDB directly through `ParquetQueryEngine`, while non-`duckdb`
  builds retain the explicit DataFusion baseline and the shared discovery or
  logical-view fallback remains separate.
- **Follow-up**: [DataFusion Query-Core Naming Landed] The surviving shared
  SQL fallback no longer presents itself as a generic request-scoped query
  core. `SearchQueryService`, the shared SQL registration path, and FlightSQL
  discovery now name that owner line explicitly as a DataFusion query core,
  which matches the residual architecture after routed Parquet execution moved
  to DuckDB.
- **Follow-up**: [FlightSQL Discovery Schema Cutover Landed]
  `CommandGetTables(include_schema = true)` no longer asks the request-scoped
  DataFusion core for a second table lookup. FlightSQL now rebuilds discovery
  schemas from `SqlQuerySurface.columns`, which keeps the public discovery
  payload stable while shrinking the residual DataFusion owner line to
  discovery registration only.
- **Follow-up**: [FlightSQL Discovery Surface Cutover Landed]
  `CommandGetDbSchemas` and `CommandGetTables` now build one request-scoped
  `SqlQuerySurface` directly from publication metadata and logical-view
  contracts, so FlightSQL discovery itself no longer opens the residual
  DataFusion query core.

## 2. [ACTIVE] Rust AST Flight Cutover

- **Task**: Remove the remaining Rust-local Julia and Modelica AST ownership
  from the active Wendao gateway path and route Studio `code_search` through
  the Julia-owned native parser line.
- **Status**: [Gateway Native Route Audit] `xiuxian-wendao-modelica/` is
  retired from the active Rust ownership line, `xiuxian-ast` no longer exports
  Julia or Modelica tree-sitter surfaces, and Studio `intent = "code_search"`
  now runs through a source-owned gateway handler plus shared
  `src/search/repo_search/` execution for both plain Julia and plain Modelica
  plugin repositories. The focused host proofs
  `search_intent_routes_code_search_to_plain_julia_plugin_repository` and
  `search_intent_routes_code_search_to_plain_modelica_plugin_repository` are
  green against the linked native parser-summary services after the
  `WendaoSearch.jl` gRPC bootstrap fix. The same linked host validation now
  also covers the repo-aware `analysis/code-ast` loader through
  `load_code_ast_analysis_response_supports_plain_julia_plugin_repository`
  and `load_code_ast_analysis_response_supports_plain_modelica_plugin_repository`,
  so both Studio `code_search` and Studio `code_ast` now ride the Julia-owned
  native parser line for plain Julia and plain Modelica repositories. The
  `xiuxian-wendao-builtin` registry bootstrap now also links that shared Julia
  plus Modelica plugin line by default instead of treating it as a feature-gated
  second bundle. The focused Studio `code_search` unit proofs for plain Julia
  and plain Modelica repositories now also run on the default test surface
  instead of hiding behind a Julia-only test gate. The linked
  `studio_repo_sync_api` Modelica proofs now follow that same rule too:
  `external_modelica_plugin_path` coverage for repo overview, module search,
  repo index, repo sync, projected pages, planner, gap reports, and symbol
  search now passes on the default unit-test surface against the Julia-owned
  native parser routes, with repo-index concurrency drift normalized in the
  test seam and the Modelica symbol snapshot updated to the current native
  parser span contract.
  The repo-backed Flight analysis route-wiring proofs now also run on the
  default unit-test surface after the shared repo fixture was tightened to
  bootstrap the linked Julia parser-summary service explicitly instead of
  falling back to the generic solver-demo base URL. The same default-surface
  rule now covers the focused repo-intelligence integration proofs for
  `repo_symbol_search` and `repo_overview`, so the Modelica gateway-search
  slice is no longer hidden behind a Julia-only test gate. The mounted
  parser-summary service itself is now also formalized on the Julia side:
  `WendaoSearch.jl` owns `config/live/parser_summary.toml` plus
  `scripts/run_parser_summary_service.jl`, and the Rust helper or transport
  surfaces now point at that package-owned parser-summary contract instead of
  describing the same mounted native-parser lane as a solver-demo service.
  The same lane now also has one formal repo-process owner path:
  `process.nix` boots `wendaosearch-parser-summary`, and the shared Rust
  parser-summary fixture can switch onto that managed service under
  `RUN_PROCESS_MANAGED_WENDAOSEARCH_TEST=1` for focused gateway `code_search`
  and `code_ast` proofs.

## 3. [ACTIVE] Wendao Vector Boundary Split

- **Task**: Separate lightweight Arrow/DataFusion substrate ownership from
  Lance-backed vector-store ownership so downstream consumers such as
  `xiuxian-qianji` do not compile `lance` unless they explicitly need
  vector-store behavior.
- **Status**: [Slice In Progress] The current bounded slice is establishing a
  dedicated architecture track for this boundary. The governing blueprint is
  the active Wendao vector-boundary split lane, and the paired ExecPlan is
  driving one first code slice that extracts lightweight substrate helpers,
  adds an explicit Wendao `vector-store` feature, and retargets Qianji to the
  non-vector Wendao surface. The baseline proof before implementation is the
  current normal dependency path
  `xiuxian-qianji -> xiuxian-wendao -> xiuxian-vector -> lance`.
