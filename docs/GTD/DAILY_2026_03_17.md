## 🌌 Core Features: DeepSeek OCR Metal Stabilization

### 1. [ACTIVE] Feature: DeepSeek Metal MoE Fast Backend

- **Task**: Continue the `metal_fast` MoE backend investigation inside vendored `deepseek-ocr`
  without touching `candle-core`.
- **Status**: [ExecPlan Active] The retained accepted head is unchanged:
  `metal_fast + eager routed experts + shared_expert_f32_compute = false`. A new pure accepted-head
  `phase=load --max-rss=13` probe is now captured and changes the immediate diagnosis: pure load
  does not pass `13 GB` either, and it dies earlier than the current `13 GB infer` evidence,
  inside layer `4` `shared_experts` load. That means the remaining target-budget miss is not only
  infer-side overhead and not only routed-expert fetch order; shared-expert load residency is back
  on the active path. After aligning the existing `skip_shared_experts` diagnostic so it also
  short-circuits load-time `shared_experts`, the `13 GB` wall still remains: both `load` and
  `infer` now die in layer `4` routed-expert eager loads instead. That narrows the next step back
  to routed-expert fetch/materialization order, while keeping shared-expert residency as a measured
  contributor rather than the full wall. A final best-case attribution check is now in place too:
  with the same `skip_shared` shape, but only `lazy_moe_experts = true`, `load 13 GB` passes at
  about `6.62 GB` RSS and a matching no-trace `infer 13 GB` repro stayed under budget for `86s`
  with observed RSS no higher than about `7.57 GB`. That makes eager routed-expert residency the
  dominant remaining wall. The newest retained landing keeps the canonical profile unchanged at the
  surface, but changes `metal_fast` internally so routed experts use a deferred load strategy when
  a deferred source exists and no snapshot is present. Under that retained implementation,
  `phase=load --max-rss=15` now passes cleanly and reaches `weights_ready` in about `14.1s`, while
  `phase=infer --max-rss=13` reaches `weights_ready` and only then dies inside
  `block.forward.moe` at around layer `3` routed-expert forward materialization (`~13.02 GB`,
  `~40.4s`). That follow-up landing is now in place too: deferred routed experts keep projection
  weights view-backed during materialization instead of forcing `gate/up/down` contiguous up front.
  Under the unchanged canonical profile, widened `phase=load --max-rss=15` still passes and the
  guarded no-trace `phase=infer --max-rss=13` now stays under budget entirely, peaking around
  `9.19 GB`. The remaining failure is no longer memory; inference completes in about `32.7s` and
  then reports `OCR produced no output`. That empty-output trace pinned the blocker to whitespace
  token `6776`. The retained follow-up now narrows first-token selection so whitespace-only
  candidates are skipped and `eos` is deferred until visible candidates are exhausted. Under the
  unchanged canonical profile, guarded `infer` now passes with `status=0` at both `13 GB` and
  `12 GB`. The retained semantic follow-up uses prompt intent for the one-digit smoke, so
  single-digit visible tokens are preferred over other visible text when the prompt explicitly asks
  for exactly one visible digit. The canonical `12 GB` run now returns `0`, and the captured
  step-0 logits confirm that this is a selection-side change: whitespace token `6776` still ranks
  first and `eos` still ranks second in the raw logits. This closes both runtime stability and the
  minimum semantic contract for the current smoke path, and the canonical profile now encodes that
  contract directly via `expected_substring = "0"`. The next exploratory quality step is now also
  characterized: a stronger amount-value probe on the same accepted `metal_fast` shape stayed
  inside the `12 GB` guard but took about `252s` and returned `No units.` instead of an amount
  substring. Two shorter month-value probes also stayed under the same `12 GB` guard, but neither
  completed in a practical smoke window, and enabling decode cache did not make that gate fast
  enough. The retained follow-up now makes one shorter stronger gate practical: a prompt-aware
  `Telegram` word probe on the same accepted head stays within the same `12 GB` budget and
  returns `Telegram` in about `37s`. Invoice suffix `001` still falls into the old long-tail
  pattern and remains rejected. The next step should therefore move from minimal one-digit smoke
  to this stronger short-word gate under the same budget, not more Metal memory work.
